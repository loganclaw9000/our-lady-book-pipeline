Expose the Phase 4 DAG orchestrator + ablation harness as CLI subcommands that land on the `book-pipeline` umbrella:

1. **`book-pipeline chapter <chapter_num>`** — composition root for the full Phase 4 loop. Mirrors the Plan 03-07 `cli/draft.py` pattern: load 4 typed configs → build LLM client via factory → construct 5 retrievers via `build_retrievers_from_config` → construct ChapterDagOrchestrator with all 11 DI components → call `orchestrator.run(chapter_num, expected_scene_count=EXPECTED_SCENE_COUNTS[chapter_num])`. Exit codes map to terminal states: 0 = DAG_COMPLETE, 2 = gate fail (missing scenes), 3 = CHAPTER_FAIL (chapter critic rejected), 4 = DAG_BLOCKED (post-commit DAG step failed), 5 = unreachable.

2. **`book-pipeline chapter-status [<chapter_num>]`** — read-only CLI. No args: print `.planning/pipeline_state.json` view. With chapter_num: print `drafts/chapter_buffer/ch{NN:02d}.state.json`'s state + dag_step + blockers + history summary.

Validates `variant-a` and `variant-b` configs load via `ModeThresholdsConfig` (SHA-hashes each via `xxhash.xxh64(path_bytes)`); computes `corpus_sha` from `indexes/resolved_model_revision.json`; pulls `voice_pin_sha` from `config/voice_pin.yaml`; constructs an `AblationRun` with a UTC timestamp run_id; calls `create_ablation_run_skeleton`; prints "runs/ablations/{run_id}/ skeleton created — Phase 6 TEST-03 will drive execution." 4. **`book_pipeline.book_specifics.outline_scene_counts.EXPECTED_SCENE_COUNTS`** — a `dict[int, int]` table giving the expected scene count per chapter. Populated from the outline structure (per outline.md OLoC real structure, or the 27×3×3 nominal). The table is the gate-check source for `orchestrator.run(expected_scene_count=...)`.
