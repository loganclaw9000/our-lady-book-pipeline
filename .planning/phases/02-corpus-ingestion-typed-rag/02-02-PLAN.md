<yaml_policy_note>
**W-4 revision (supersedes STACK.md's "ruamel.yaml rejected because we don't round-trip YAML" decision):**

The original plan had CorpusIngester write the resolved BGE-M3 SHA back into `config/rag_retrievers.yaml`. That forced either (a) use ruamel.yaml (rejected by STACK.md), or (b) use PyYAML safe_dump and DESTROY all comments in a git-tracked config file — unacceptable.

The fix is to eliminate the YAML round-trip entirely. Resolved SHA lives in its own JSON file outside git:

- `indexes/resolved_model_revision.json` (created on first successful ingest; gitignored)
- Shape: `{"sha": "abc123...", "resolved_at": "2026-04-21T02:14:33Z", "model": "BAAI/bge-m3"}`
- CorpusIngester writes this AFTER successful embedding-model load; the value flows FROM the embedder back into future ingest runs.
- When loading the embedder, the ingester checks: if `config/rag_retrievers.yaml`'s `embeddings.model_revision` is `"TBD-phase2"` or `"latest-stable"` or matches `^TBD-.*`, load `indexes/resolved_model_revision.json` and pass its `sha` as the revision to `BgeM3Embedder(revision=sha, ...)`. If the JSON doesn't exist yet (first run), pass `revision=None` so the embedder resolves from HF; persist the result.
- `config/rag_retrievers.yaml` is READ-ONLY to the ingester. Human edits only.
- `.gitignore` gains the new file pattern.

This honors the STACK.md decision by removing the round-trip requirement entirely, rather than picking a round-trip library.
</yaml_policy_note>
