<outline_format_notes>
Expected outline.md format (per 02-CONTEXT.md "27 chapters × blocks × beats"):

```
# Chapter 1: [Title]
## Block A: [Block name]
### Beat 1: [Beat function description]
[beat content...]
### Beat 2: [Beat function description]
...
## Block B: [Block name]
### Beat 1: ...

# Chapter 2: [Title]
...
```

Beat ID schema: `ch{chapter_num:02d}_b{block_letter_or_num}_beat{beat_num:02d}`
- chapter from `# Chapter N:` heading (regex `^#\s+Chapter\s+(\d+)`)
- block from `## Block X:` heading (regex `^##\s+Block\s+(\w+)`) — lowercase + strip non-alphanumeric
- beat from `### Beat N:` heading (regex `^###\s+Beat\s+(\d+)`)

Stability rule: beat ID = `f"ch{chapter:02d}_b{block_id.lower()}_beat{beat:02d}"`. Stays stable across re-parses as long as the chapter/block/beat NUMBERING is unchanged. If beat CONTENT text changes, ID is unchanged (this is the feature — scene requests reference beat IDs; content may be edited without invalidating cross-references).

If outline.md doesn't follow this format exactly (likely for real OLoC outline), the parser should be LENIENT: attempt several regex patterns, log warnings for unrecognized headings, but never crash. The parse result is `list[Beat]` with fields `{beat_id, chapter, block, beat, title, body, heading_path}`; `body` is the markdown under the Beat heading up to the next Beat/Block/Chapter heading.
</outline_format_notes>
